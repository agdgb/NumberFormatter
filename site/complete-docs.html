Getting Started

HumanNumbers is a formatting pipeline that ensures every number your application emits is immediately understandable to humans. Instead of scattering formatting logic across controllers, views, serializers, and JavaScript, HumanNumbers centralizes numeric presentation into a single, culture‑aware system integrated throughout the ASP.NET Core stack.

When Should You Use HumanNumbers?

HumanNumbers operates at multiple layers of the ASP.NET Core stack so formatting becomes automatic rather than repetitive. Typical usage includes:

Use HumanNumbers when:

  • + Financial APIs returning pre‑formatted values
  • + Razor views that auto‑format numeric models
  • + Dashboards that display compact metrics consistently
  • + Multi‑culture applications requiring per‑request formatting

Do NOT use when:

  • - Machine-to-machine APIs
  • - Scientific/engineering calculations
  • - Raw data persistence

Why Not Built-in .NET Formatting?

Built-in .NET formatting is designed for precision and machine readability. HumanNumbers is optimized for human readability and performance in UI scenarios, providing compact notation, zero-allocation operations, and guaranteed non-throwing behavior.

Request Culture Resolution

HumanNumbers determines formatting culture using the ASP.NET Core request pipeline, ensuring consistent formatting across all layers of your application.

Resolution Priority

  1. 1. Explicit per‑request override - Manual culture setting per request
  2. 2. RequestLocalization middleware culture - Middleware-determined culture
  3. 3. Current thread culture - Fallback to thread culture
// Manual culture override for specific scenarios
CultureResolver.SetCulture(HttpContext, new CultureInfo("fr-FR"));

// This allows endpoints to force culture when required
// For example: exporting reports in a specific language

Why Pipeline Resolution Matters

Automatic culture resolution means your APIs and views adapt to user preferences without manual formatting code. However, you retain full control to override when business logic requires specific formatting (e.g., financial reports in a fixed currency locale).

Why HumanNumbers?

Traditional number formatting in .NET allocates memory for every formatting operation. HumanNumbers uses Span<char> and stackalloc to provide zero-allocation formatting, making it ideal for high-performance applications like financial systems and real-time dashboards.

Quick Start

Start formatting numbers with just a few lines of code: 

using HumanNumbers;

// Basic number formatting
var result = 1500000.ToHuman();        // "1.5M"
var currency = 1500000.ToCurrency();   // "$1.5M"

// Financial words for check printing
var words = 1500.ToFinancialWords();   // "One Thousand Five Hundred"

// Basis points for financial markets
var bps = 0.015.ToBasisPoints();       // "150 bps"

// Byte scaling for technical applications
var bytes = 1048576.ToBytes();         // "1.0 MB"

Performance Note

All formatting operations are zero-allocation when the result fits in the stack buffer. For larger numbers, only a single allocation occurs for the result string.

The foundation layer containing all number formatting logic and performance optimizations.

Middleware and integration components for seamless ASP.NET Core adoption.

  • JSON serialization converters and factories

  • Error handling and validation feedback
  • Localization and internationalization support
  • Real-time dashboard and analytics integration

Choosing a Configuration Approach

HumanNumbers provides three layers of configuration to meet different needs. Understanding when to use each approach ensures clean, maintainable code:

1. Global Defaults

Set application-wide behavior during startup. Use for consistent formatting across your entire application. 

// Program.cs - Global configuration
builder.Services.AddHumanNumbersDefaults();

// Or with custom defaults
builder.Services.AddHumanNumbersCore(options => 
{
    options.DefaultCulture = new CultureInfo("en-US"),
    options.ErrorMode = HumanNumbersErrorMode.Safe,
    options.EnableCaching = true
});

When to use: Most applications need consistent formatting rules across all endpoints and views.

2. Policies & Attributes

Define scenario-specific behavior through policies and model attributes. Use for different formatting needs in different contexts. 

// Model attributes for JSON serialization
public class FinancialReport
{
    [HumanNumber(OutputMode = HumanNumberOutputMode.SerializeAsHuman)]
    public decimal Revenue { get; set; }
    
    [HumanNumber(IsCurrency = true, DecimalPlaces = 2)]
    public decimal Expenses { get; set; }
}

// Policy-based configuration
builder.Services.AddHumanNumbersCore(options => 
{
    options.Policies["Financial"] = new HumanNumberFormatOptions 
    { 
        DecimalPlaces = 2, 
        UseMinPrecision = false 
    };
    options.Policies["Technical"] = new HumanNumberFormatOptions 
    { 
        DecimalPlaces = 1, 
        UseMinPrecision = true 
    };
});

When to use: Different APIs or views need different formatting rules (financial vs technical, public vs internal).

3. Per-Call Options

Override behavior for specific formatting operations. Use for one-off formatting needs or dynamic requirements. 

// Manual formatting with custom options
var result = value.ToHuman(new HumanNumberFormatOptions 
{ 
    DecimalPlaces = 3, 
    Culture = new CultureInfo("fr-FR") 
});

// Service-based formatting
var formatter = serviceProvider.GetRequiredService();
var formatted = formatter.Format(value, "FinancialPolicy");

// Span-based formatting for maximum performance
Span buffer = stackalloc char[32];
var success = value.TryFormatHuman(buffer, out var written, out var result);

When to use: Specific scenarios need different formatting than global or policy-based settings.

Configuration Hierarchy

The three layers follow a clear hierarchy: Per-call options override policies, which override global defaults. This allows you to start with sensible global behavior and selectively override where needed, keeping your code DRY while maintaining flexibility.

        var portfolio = _service.GetPortfolio(id);
        
        // Zero allocation JSON formatting
        return Ok(new
        {
            portfolio.Name,
            TotalValue = portfolio.TotalValue.ToHuman(),
            Positions = portfolio.Positions.Select(p => new
            {
                p.Symbol,
                Value = p.Value.ToHuman(),
                Change = p.Change.ToBasisPoints()
            })
        });
    }
